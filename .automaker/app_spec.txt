  <project_name>Mesh ARKade</project_name>

  <overview>
    Mesh ARKade is a decentralized game preservation platform built entirely on the Pear Runtime (Holepunch ecosystem). It operates as a CLI-first, P2P-native &quot;museum-quality&quot; archive for retro game ROMs. The application provides a headless terminal interface for managing game library mounts, discovering supported retro systems via Libretro&apos;s DAT database, seeding game metadata (wishlist) from No-Intro/Redump DAT files into a local Hyperbee database, searching the seeded catalog, and fetching ROMs via a triple-layer P2P discovery stack (Hyperswarm → IPFS → BitTorrent DHT). The architecture follows a singleton CoreHub pattern with JSON-RPC style request dispatch, a dual-runtime abstraction layer (Bare runtime vs Node.js), and a modular CLI with individually testable command handlers. A React/Vite GUI mode exists as a secondary interface (Pear Electron) but the primary focus is the headless CLI where both human users and AI agents are equal citizens. The project uses an OpenSpec workflow for change management and enforces SOLID, DRY, and TDD principles.
  </overview>

  <development_guidelines>
    <guideline>SOLID principles enforced: Single responsibility (one handler per file), open/closed (CommandHandler interface), Liskov substitution (CoreHub interface for testing), interface segregation (minimal ReadlineInterface), dependency inversion (runtime abstraction layer)</guideline>
    <guideline>DRY: Shared formatter module, centralized runtime resolution, reusable parser</guideline>
    <guideline>TDD: Tests written alongside implementation, 80% coverage thresholds enforced via vitest Istanbul</guideline>
    <guideline>Terminal-first: CLI is the primary interface; GUI is secondary. All core logic must be headless-compatible</guideline>
    <guideline>Bare Runtime compatibility: No DOM or Node-specific assumptions in core logic. Use runtime.ts abstraction for fs/path/os/fetch/crypto</guideline>
    <guideline>Pear Guardian: Non-Pear-native solutions (HTTP servers, localStorage, standard WebSocket) are flagged with &apos;Timeout! ⏰&apos;</guideline>
    <guideline>TSDoc comments must explain intent and guarantees, never restate implementation. Shape: Intent → Guarantees → Constraints/Warnings</guideline>
    <guideline>Git discipline: Never use &apos;git add .&apos;, never force push, explicit file staging</guideline>
    <guideline>No &apos;any&apos; types in production code (runtime.ts caches exempted with documented rationale)</guideline>
    <guideline>OpenSpec workflow for all changes: proposal → design → specs → tasks → implementation → archive</guideline>
    <guideline>Agent-friendly: All commands support --json flag for machine-parseable output. Agents and humans are equal CLI users</guideline>
    <guideline>Pre-commit hooks: vitest related tests + TSDoc checker on staged files</guideline>
  </development_guidelines>
