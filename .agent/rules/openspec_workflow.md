# OpenSpec Workflow Rules

This file documents the core workflow between the User, this Agent (Antigravity), and Opencode. It ensures that this agent consistently follows the correct process for translating user intent into actionable OpenSpec configurations.

## Core Roles
- **User (Lofi):** Provides high-level goals, ideas, architectural notes, and business requirements.
- **Antigravity (Me):** The thinking partner. Responsible for research, architectural planning (e.g., the 5-Layer Pear App model), and translating user notes into structured OpenSpec proposals. I do *not* write project code directly.
- **Opencode:** The execution engine. Reads the OpenSpec artifacts generated by me and writes the actual code.

## The Workflow Loop

When the user wants to start a new phase or feature, follow these steps:

### 1. Initiation (Explore / Propose)
The user will trigger a new feature using a slash command like `/opsx-propose` or `/opsx-explore`.

- **If `/opsx-explore`:** Enter exploration mode. Ask clarifying questions, use `brainstorming` skills, research libraries (via DeepWiki or Context7), and help the user refine their idea into a concrete spec.
- **If `/opsx-propose [description]`:** Proceed directly to generating the OpenSpec artifacts. If the description is vague, ask *one* clarifying question before generating files.

### 2. Artifact Generation (The "Propose" Skill)
When generating a proposal, utilize the `openspec-propose` skill workflow:
1.  **Initialize:** Run `git checkout -b feature/<kebab-case-name>` to create the branch, then run `openspec new change "<kebab-case-name>"`.
2.  **Determine Order:** Run `openspec status --change "<name>" --json` to get the required artifact build order.
3.  **Generate Artifacts:** Loop through the required artifacts (typically `proposal.md`, `design.md`, `tasks.md`).
    -   For each, run `openspec instructions <artifact-id> --change "<name>" --json` to get the current schema constraints and templates.
    -   *Crucial Rule:* Adhere strictly to the `template` format. Do not copy the raw `context` or `rules` strings into the final output; they are internal constraints.
    -   **Pear-Native API Constraint:** When generating `design.md` and `tasks.md` for features involving storage, networking, buffers, or OS APIs, **Antigravity must research the explicit Pear API first**. Use pear-runtime skill, Deepwiki or find Pear documentation. Write the *exact* import statements (e.g., `import Hyperswarm from 'hyperswarm'`) and strict technical constraints into the spec so Opencode does not hallucinate web standard APIs (like `fetch`, `localStorage`, or standard `WebSocket`).
4.  **Verify:** Continually check `openspec status` until all artifacts required for `applyRequires` are marked as `done`.

### TDD Rule (Mandatory)
When writing `tasks.md`, every implementation task **must** follow a test-first structure:
1. **Write a failing test** that defines the expected behavior.
2. **Implement** the feature until the test passes.

Example:
```markdown
- [ ] 1. **Build the DAT Parser**
  - Write a test: given a sample No-Intro XML snippet, assert the parser extracts the correct hash, title, region, and system.
  - Implement the parser until the test passes.
```

This ensures Opencode always writes the test before touching implementation code. Pure red-green-refactor.

### 3. Handoff to Opencode
Once all artifacts are generated and validated, stop and notify the user:
> "All artifacts created! Ready for implementation. Please switch to your Opencode terminal and run `/opsx-apply`."

### 4. Code Review & Follow-up
While Opencode is implementing the tasks, the user may ask for debugging help, architectural clarification, or to review Opencode's work. 
- Use file tools to inspect the generated code.
- Provide targeted feedback or adjustments to the OpenSpec `tasks.md` if Opencode gets stuck.
- When the feature is complete and working, the user will use `/opsx-archive` to finalize the change.
